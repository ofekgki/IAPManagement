Integration guide

Integrate the Purchase SDK

Everything you need to add in-app purchases to an Android app with the Purchase SDK — from adding the dependency to gating premium features, restoring purchases, and reading analytics.

Manage products, prices, and API keys in the developer portal — the Android SDK reads them live from the backend, so you can change your catalog without shipping a new app release. Everything runs end-to-end in a simulated MOCK billing mode (no real charges).

1. Core concepts

Four things live on the backend and are managed in the portal; the SDK references them by their string ids.

ConceptWhat it is
AppA container for one game/app. Owns its API keys, catalog, purchases and analytics. Identified by an appId.
API keyThe credential the SDK sends as X-SDK-API-Key. The raw key is shown once; only a hash is stored.
ItemA product (remove_ads, coins_100, pro_monthly…). Has a type (LIFETIME/CONSUMABLE/SUBSCRIPTION), a price in minor units (499 = $4.99), and the entitlement it grants.
EntitlementProof a user owns something. You gate features by checking hasEntitlement(itemId).
PurchaseA record of a buy attempt with a status lifecycle (CREATED → PENDING → SUCCESS, or FAILED/CANCELLED/RESTORED) and a price snapshot.
Payment methodThe tender chosen at purchase: APPLE_PAY / GOOGLE_PLAY / PAYPAL / CREDIT_CARD — recorded and charted in the portal.

2. Requirements

RequirementValue
Minimum Android SDKminSdk 24 (Android 7.0)
LanguageKotlin (the public API is coroutine-based)
Host themeNone required — the popup themes itself (Material Components not needed)
BackendThe platform backend running and reachable (see step 4). Run it locally with docker compose up --build.
Internet permissionAdded automatically by the SDK's manifest
The SDK talks to your backend over HTTP. In the demo, that's http://localhost:8080 — reachable from an emulator as http://10.0.2.2:8080.

3. Add the dependency (JitPack)

  1. Add the JitPack repository

    In settings.gradle.kts (or your root build.gradle):

    dependencyResolutionManagement {
  repositories {
    google()
    mavenCentral()
    maven { url = uri("https://jitpack.io") }   // <-- add this
  }
}

  2. Add the SDK dependency

    In your app module's build.gradle.kts. Use a release tag, or main-SNAPSHOT for the latest commit:

    dependencies {
  implementation("com.github.ofekgki.IAPManagement:iap-sdk:1.0.0")
}

    The coordinate is com.github.<user>.<repo>:<module>. Browse versions at jitpack.io/#ofekgki/IAPManagement.

Working inside the monorepo instead of consuming from JitPack? Depend on the module directly: implementation(project(":iap-sdk")).

4. Create an app & API key

The SDK needs an API key and at least one item, both created in the portal. In the demo these are pre-seeded (demo_api_key_123, ~21 items).

  1. Run the backend & portal

    docker compose up --build → portal at http://localhost:5173, backend at :8080.

  2. Sign in

    demo@example.com / password123, or register a new account.

  3. Create an app

    New app → name + package (e.g. com.example.game) + default billing mode.

  4. Generate an API key

    App → API Keys → Create key. Copy the raw key now — only a hash is stored.

  5. Create an item

    App → Items → New item. Note the item id (e.g. remove_ads); price is in minor units (499 = $4.99).

5. Initialize the SDK

Pick one of the two initialization styles.

Option A — zero-config (recommended)

Declare meta-data in your manifest; the SDK's PurchaseSdkInitProvider (a ContentProvider, the same pattern Firebase uses) auto-initializes before Application.onCreate. No init code needed.

<!-- AndroidManifest.xml, inside <application> -->
<meta-data android:name="com.example.iapsdk.API_KEY"      android:value="demo_api_key_123" />
<meta-data android:name="com.example.iapsdk.BASE_URL"     android:value="http://10.0.2.2:8080/api/v1/sdk" />
<meta-data android:name="com.example.iapsdk.BILLING_MODE" android:value="MOCK" />
The user id is usually unknown at startup. Once the user signs in, call PurchaseSdk.init(context, apiKey, userId = currentUser.id) to attach it.

Option B — manual init

Prefer explicit control (or have a user id up front)? Call init in Application.onCreate:

class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        PurchaseSdk.init(
            context = applicationContext,
            apiKey = "demo_api_key_123",
            billingMode = BillingMode.MOCK,
            userId = "user-123",
            config = PurchaseSdkConfig(baseUrl = "http://10.0.2.2:8080/api/v1/sdk"),
        )
    }
}
ParameterMeaning
apiKeyThe raw API key from the portal (sent as X-SDK-API-Key).
billingModeMOCK (simulated) or GOOGLE_PLAY (scaffolded, fails safe until configured).
userIdYour app's user id. Optional at init; set it when the user signs in.
config.baseUrlYour backend's SDK API base. Emulator → http://10.0.2.2:8080/api/v1/sdk.

6. Show the purchase popup

The drop-in bottom sheet renders per-product artwork and an in-popup payment-method picker, then drives the whole start → confirm flow for you.

PurchaseSdk.showPurchasePopup(
    activity = this,
    itemId = "remove_ads",
    listener = object : PurchaseListener {
        override fun onPurchaseSuccess(result: PurchaseResult) { unlockFeature() }
        override fun onPurchaseCancelled() { /* user dismissed */ }
        override fun onPurchaseFailed(error: PurchaseSdkError) { toast(error.message) }
    },
)
Emulator note: 10.0.2.2 is the emulator's alias for your computer's localhost. On a physical device, use your machine's LAN IP.

7. Gate features with entitlements

Entitlement checks are network-first, so revokes, expiries, and returns are reflected immediately.

lifecycleScope.launch {
    if (PurchaseSdk.hasEntitlement("remove_ads")) showPremiumUi() else showPaywall()
}

// List everything the user owns:
val owned: List<UserEntitlement> = PurchaseSdk.listEntitlements()

8. Restore & return

Restore returns owned items (a refund in this platform): the item becomes buyable again and its price is netted out of revenue. Restore everything, or a single item.

PurchaseSdk.restorePurchases()              // return everything currently owned
PurchaseSdk.restorePurchases("remove_ads")  // return a single item

Each returned purchase is recorded with status RESTORED so the portal can chart returns separately from sales.

9. Headless purchase & payment methods

Building your own purchase UI instead of the popup? Load the catalog and call makePurchase directly, passing the chosen payment method.

val items: List<PurchaseItem> = PurchaseSdk.getItems()       // suspend
val item: PurchaseItem = PurchaseSdk.getItem("remove_ads")   // suspend

val result = PurchaseSdk.makePurchase("remove_ads", paymentMethodId = "PAYPAL")
Payment method idShown as
APPLE_PAYApple Pay
GOOGLE_PLAYGoogle Pay
PAYPALPayPal
CREDIT_CARDCredit Card (default)

10. Analytics events

Views, popup opens, purchases, returns, and entitlement checks are tracked automatically and stream to the portal. Emit your own events too:

PurchaseSdk.trackEvent("paywall_viewed", mapOf("source" to "settings"))

These power the portal's funnel, revenue, and purchases-by-status charts.

11. Error handling & fail-safe

The suspend APIs throw PurchaseException; the popup reports through onPurchaseFailed. Match on the typed PurchaseSdkError.

try {
    PurchaseSdk.makePurchase("remove_ads")
} catch (e: PurchaseException) {
    when (e.error) {
        is PurchaseSdkError.GooglePlayNotConfigured -> fallBackToMock()
        is PurchaseSdkError.NotInitialized -> initThenRetry()
        else -> showError(e.error.message)
    }
}
Fails safe: GOOGLE_PLAY mode never fakes a successful verification — it returns GooglePlayNotConfigured / REQUIRES_VERIFICATION until real server-side verification is wired up.

13. Troubleshooting

SymptomCause & fix
INVALID_API_KEYWrong/rotated key, or app disabled in the portal. Re-copy the raw key; re-enable the app.
NotInitializedinit hasn't run. Add the manifest meta-data (Option A) or call init before any SDK call.
Requests time out on device10.0.2.2 only works on the emulator. On a physical device set baseUrl to your machine's LAN IP.
Purchase stuck at CREATEDStarted but never confirmed. The SDK uses a unique idempotency key per attempt; check the purchase's event log in the portal.
GOOGLE_PLAY_NOT_CONFIGUREDExpected — Google Play verification is scaffolded. Use MOCK mode for demos.
